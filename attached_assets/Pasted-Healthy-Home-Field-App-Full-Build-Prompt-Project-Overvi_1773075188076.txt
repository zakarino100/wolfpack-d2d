Healthy Home Field App — Full Build Prompt
Project Overview
Rebuild and extend the existing Healthy Home door-to-door sales rep app into a production-ready V1. This is a React Native app (Expo bare workflow) with an Express.js + TypeScript backend connected to Supabase (PostgreSQL). The app serves two user types: Reps (field canvassers) and Admins (owners/managers).

Tech Stack
Frontend: React Native, Expo bare workflow, TypeScript
Backend: Express.js, TypeScript, tsx, port 5000
Database: Supabase (PostgreSQL)
Auth: Google OAuth + email/password, JWT tokens
Maps: React Native Maps + Google Maps API
Offline: AsyncStorage queue — writes saved locally, synced to Supabase when signal returns
Push Notifications: Expo Notifications
SMS: React Native's Linking API (opens native SMS composer)
Styling: NativeWind (Tailwind for React Native)

Database Schema (Supabase)
Create these tables:
sql-- Users
users (
  id uuid primary key,
  email text unique,
  name text,
  role text check (role in ('rep', 'admin')),
  google_id text,
  avatar_url text,
  created_at timestamptz default now()
)

-- Leads
leads (
  id uuid primary key,
  rep_id uuid references users(id),
  address text not null,
  lat float not null,
  lng float not null,
  status text check (status in ('not_home', 'not_interested', 'follow_up', 'sold', 'completed')),
  services_interested text[], -- ['house_wash', 'cement_cleaning', etc]
  contact_name text,
  contact_phone text,
  notes text,
  follow_up_date timestamptz,
  -- Scheduling preferences captured during sale
  preferred_day text, -- 'monday', 'tuesday', etc
  preferred_time text, -- 'morning', 'afternoon', 'evening'
  scheduling_notes text,
  sold_at timestamptz,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
)

-- Routes
routes (
  id uuid primary key,
  admin_id uuid references users(id),
  rep_id uuid references users(id),
  name text,
  date date not null,
  status text check (status in ('draft', 'shared', 'in_progress', 'completed')),
  created_at timestamptz default now()
)

-- Route stops (jobs on a route, ordered)
route_stops (
  id uuid primary key,
  route_id uuid references routes(id) on delete cascade,
  lead_id uuid references leads(id),
  stop_order int not null,
  arrival_window text, -- e.g. '9am-11am'
  notes text -- admin notes for this stop only
)

-- Rep GPS tracking
rep_locations (
  id uuid primary key,
  rep_id uuid references users(id),
  lat float,
  lng float,
  recorded_at timestamptz default now()
)

-- Offline sync queue (local only, not Supabase)
-- Stored in AsyncStorage as JSON array of pending mutations
```

## Authentication

Support two login methods:

**Google OAuth:**
- Uses existing Google OAuth flow with JWT
- On first login, create user record with role = 'rep' by default
- Admin role must be manually set in Supabase (no self-service admin)

**Email + Password:**
- Standard email/password with Supabase Auth
- Same role logic applies

**JWT middleware** on all protected Express routes checking role.

**On app launch:**
- If no token → Login screen
- If token + role = rep → Rep app
- If token + role = admin → Admin dashboard

- If no signal, pin drops and Lead Form saves to AsyncStorage queue
- A banner shows "Offline — X items pending sync"
- When signal returns, queue auto-syncs to Supabase in background
- Synced pins replace local queue items

**Rep's own location:**
- Blue pulsing dot showing current position
- Location updates POST to `rep_locations` table every 60 seconds while app is foregrounded

Offline Architecture
AsyncStorage queue structure:
typescriptinterface OfflineQueueItem {
  id: string; // local temp ID
  type: 'create_lead' | 'update_lead' | 'update_stop_status';
  payload: object;
  created_at: string;
  synced: boolean;
}
```

- All mutations go through a `useOfflineQueue` hook
- Hook checks network status (NetInfo)
- If online: POST directly to API, mark synced
- If offline: save to AsyncStorage queue
- Background sync: NetInfo `addEventListener` triggers queue flush on reconnect
- Conflict resolution: server timestamp wins for updates; local creates always push through

## API Routes (Express)
```
POST   /auth/google          → Google OAuth
POST   /auth/login           → email+password login
POST   /auth/register        → email+password register
GET    /auth/me              → current user

GET    /leads                → all leads (admin) or own leads (rep)
POST   /leads                → create lead
PUT    /leads/:id            → update lead
DELETE /leads/:id            → delete lead (admin only)

GET    /routes               → routes (admin: all, rep: assigned to them)
POST   /routes               → create route (admin only)
PUT    /routes/:id           → update route (admin only)
POST   /routes/:id/stops     → add stop to route
PUT    /routes/:id/stops/:stopId → update stop order/status
DELETE /routes/:id/stops/:stopId → remove stop

POST   /rep-locations        → update rep GPS position
GET    /rep-locations/live   → all active rep locations (admin only)

GET    /analytics/summary    → dashboard stats
GET    /analytics/reps       → per-rep breakdown

GET    /users                → all users (admin only)
PUT    /users/:id/role       → change role (admin only)
DELETE /users/:id            → deactivate user (admin only)
```

All routes protected by JWT middleware. Role-checked where noted.

Key Rules

Never expose pricing in any SMS route share
Admin can see and edit all data; reps can only see and edit their own leads
Location sharing is opt-in for reps — never force it
Offline queue must never lose data — if sync fails, retry with exponential backoff
All sold lead scheduling preferences must be visible in route builder
Route SMS share must use native SMS composer via Linking API — no third party SMS service needed
