## DATA QUALITY RULES FOR THIS CSV
--------------------------------------------------

Handle these known issues:

1. Some records have no last name
- import as empty string or null based on current schema conventions

2. Some records have no address
- import anyway
- geocoding should fail gracefully
- row should still be available in admin even if it cannot appear on map without coordinates

3. Phone numbers are inconsistent
- examples: (919) 555-1234, +15551234567, raw digits
- store raw input in phone_raw
- normalize to E.164 in phone_normalized where possible

4. Total Quote has $ prefix
- strip before parsing decimal

5. Purchased and Serviced contain "checked" or empty
- parse accordingly

6. Some addresses contain embedded newlines
- normalize to one line for storage/geocoding

7. Some rows contain nonsense test data
- import anyway unless clearly invalid according to critical field rules

8. Duplicate customer entries exist
- use the dedupe strategy above
- keep the first or most complete existing record
- do not create visible duplicates
